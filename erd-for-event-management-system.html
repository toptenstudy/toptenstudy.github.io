2. is it easy to represent data using ERD?​


Database troubleshooting: ER diagrams are used to analyze existing databases to find and resolve problems in logic or deployment. Drawing the diagram should reveal where it's going wrong. Business information systems: The diagrams are used to design or analyze relational databases used in business processes.

What characteristics of data are represented in an E-R diagram? An E-R diagram shows many characteristics of data, including the definition, structure, and relationships within data. Additionally, this diagram shows cardinalities, relationship degrees, and business rules.

Graphical representation for better understanding: An E-R model gives graphical and diagrammatically representation of various entities, its attributes and relationships between entities. This is turn helps in the clear understanding of the data structure and in minimizing redundancy and other problems.

CROW'S FOOT ERD USING SPECIALIZATION HIERARCHY

Crow's Foot ERD (Entity-Relationship Diagram) is a graphical representation of entities and their relationships to each other in a database. It is a popular and widely used tool for data modeling, which helps to visualize and organize the various entities, attributes, and relationships in a database system. In a Crow's Foot ERD, the entities are represented as rectangles, and the relationships between entities are represented as lines connecting the rectangles. The lines are annotated with symbols to indicate the type of relationship between entities, such as one-to-one, one-to-many, or many-to-many relationships. The Crow's Foot notation is named after the crow's foot symbol that represents a one-to-many relationship.

Sample Crow's foot ERD

     +--------------+                        +--------------+

     |  Department  |                |    Project   |

     +--------------+                +--------------+

     |  Dept_ID (PK)|                | Project_ID (PK) |

     |  Name        |                | Title          |

     |  Max_Emps    |                | Budget         |

     |  Manager_ID (FK) ----------<>|  Manager_ID (FK)|

     +--------------+                +--------------+

                            |

                            |

                   +------+------+

                     |             |

     +--------------+    +-------------------+

     | Researcher   |    | Administrative     |

     +--------------+    |       Assistant          |

     | Researcher_ID|    +-------------------+

     | Name         |    | Assistant_ID (PK)  |

     | Office       |    | Name              |

     | Phone        |    | Typing_Speed      |

     | Job_Class    |    +-------------------+

     | Hire_Date    |

     | Salary       |

     | Specialty    |

     | Highest_Degree|

     +--------------+

     +--------+

     |  Manages|

     +--------+

     |  Manager_ID (PK) (FK) |

     |  Department_ID (FK) |

     |  Project_ID (FK)    |

     |  Bonus              |

     +---------------------+

     +------------+

     |  Works_On  |

     +------------+

     |  Project_ID (PK) (FK)|

     |  Researcher_ID (PK) (FK)|

     |  Hours_Spent          |

     +----------------------+

In this updated ERD, we have added three new entities: Researcher, Administrative Assistant, and Manages.

Researcher has attributes Researcher_ID, Name, Office, Phone, Job_Class, Hire_Date, Salary, Specialty, and Highest_Degree, where Researcher_ID is the primary key of the table.

Administrative Assistant has attributes Assistant_ID, Name, and Typing_Speed, where Assistant_ID is the primary key of the table.

Manages has attributes Manager_ID, Department_ID, Project_ID, and Bonus, where Manager_ID is the primary key and also a foreign key referencing Researcher.Researcher_ID. Department_ID and Project_ID are foreign keys referencing Department.Dept_ID and Project.Project_ID, respectively.

In addition, we have added a new relationship between Department and Manages, which is a one-to-many relationship representing the fact that each department can have one manager, but each manager can manage many departments.

We have also added a new relationship between Project and Manages, which is a one-to-many relationship representing the fact that each project can have one manager, but each manager can manage many projects.

Finally, we have added a new relationship between Researcher and Works_On, which is a many-to-many relationship representing the fact that each researcher can work on many projects, and each project can have many researchers working on it. The Works_On table has attributes Project_ID, Researcher_ID, and Hours_Spent, where Project_ID and Researcher_ID are foreign keys referencing Project.Project_ID and Researcher.Researcher_ID, respectively. The Hours_Spent attribute tracks the number of hours a researcher has spent working on a particular project.

Note that we have included primary keys.
